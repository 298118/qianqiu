# Session Storage Migration Plan

S38.2 began as a planning step and now also records the implemented JSON storage hardening baseline. The current implementation remains local JSON under `data/sessions/`, but it now uses a versioned storage envelope, legacy migration, atomic writes, per-session mutation serialization, revision checks, a save-list API, and an S38.3 browser save-list UI. SQLite or hosted database storage remains a future adapter step.

## Implemented JSON Baseline

- Sessions are JSON files at `data/sessions/{sessionId}.json`.
- `sessionId` is generated by `crypto.randomUUID()` and must match the UUID-like safe pattern in `src/storage/sessionStore.js` before any path is built.
- New writes persist a top-level storage envelope with `storageSchemaVersion`, `sessionId`, timestamps, `revision`, redacted metadata, and nested `worldState`.
- `writeSession(worldState)` creates `data/sessions/` and writes the envelope with same-directory temp-file-and-rename replacement.
- `readSession(sessionId)` reads and parses the JSON file, returning only route-compatible `worldState`; it returns `404` for missing safe ids and `400` for unsafe ids.
- Legacy raw `worldState` files are treated as schema `0` and migrated to the envelope on read.
- Routes normalize older feature slices after reads with helpers such as `ensureRelationshipLedger()`, `ensureExamCalendarState()`, `ensureLongTermEventState()`, `ensureOfficialCareerState()`, and `ensureRoleWorldCouplingState()`.
- Game and exam mutation routes use `mutateSession()` so overlapping writes for the same session are serialized and revision-checked.
- `GET /api/game/saves` returns redacted save-list metadata from `listSessions()`.
- Browser restore still stores `qianqiu.sessionId` in `localStorage` for compatibility, and S38.3 adds a start-page plus in-game save picker backed by `GET /api/game/saves`.

Known gaps:

- No cross-process lock for multiple Node servers sharing one directory.
- No backup/export shape or database adapter implementation yet.
- No SQLite or hosted database migration yet.

## Target Invariants

Storage changes must preserve these project invariants:

- `npm install && npm start` remains enough for local Mock-mode play.
- Real model providers remain optional and migrations never call providers.
- The server remains the authority for state boundaries, promotions, anti-cheat penalties, event queues, and persistence.
- The complete scholar path remains playable: scholar -> child exam -> provincial exam -> metropolitan exam -> palace exam -> official.
- `data/sessions/*.json`, `.env`, and provider keys stay out of Git.
- Legacy JSON saves should either migrate in memory and save back safely, or fail with an actionable error that does not corrupt the original file.

## Session Schema Versioning

The storage envelope is implemented as:

```json
{
  "storageSchemaVersion": 1,
  "sessionId": "uuid",
  "createdAt": "2026-05-06T00:00:00.000Z",
  "updatedAt": "2026-05-06T00:00:00.000Z",
  "revision": 1,
  "metadata": {
    "playerName": "未定",
    "role": "scholar",
    "roleLabel": "书生",
    "dynasty": "明",
    "year": 1644,
    "month": 1,
    "turnCount": 0,
    "examRank": null,
    "palaceRank": null,
    "officeTitle": null,
    "summary": "寒窗士子"
  },
  "worldState": {}
}
```

Compatibility rules:

- Raw legacy files are treated as storage schema `0`, where the entire parsed object is the `worldState`.
- `worldState.sessionId` remains required and must match the file id. Envelope `sessionId` must match both.
- Feature-level `schemaVersion` fields such as `examCalendar.schemaVersion` and `officialCareer.schemaVersion` stay inside `worldState`; they are not replacements for the storage envelope version.
- Migration helpers are pure at the normalization layer: `normalizeSessionRecord(parsed) -> { record, appliedMigrations }`; persistence happens only at the store boundary.
- Unknown future `storageSchemaVersion` values return a clear read error instead of silently downgrading.

Implemented migration order:

1. Added `sessionRecord` helpers that wrap legacy raw saves without changing route payloads.
2. Added tests for raw legacy read, envelope read, mismatched ids, unsupported future version, metadata rebuilding, save-list redaction, and mutation serialization.
3. Switched `writeSession()` to write the envelope while keeping `readSession()` returning `worldState` for existing route compatibility.
4. Added explicit migration notes to `docs/SHARED_CONTEXT.md`, `docs/DEVELOPMENT_STEPS.md`, README, architecture notes, and the product brief.

## Atomic Writes

JSON storage now uses a temp-write-and-rename protocol:

1. Serialize the full session record before touching the existing file.
2. Write to a unique temp file in `data/sessions/`, for example `{sessionId}.json.{pid}.{timestamp}.{uuid}.tmp`.
3. Open the temp file, write UTF-8 JSON with a trailing newline, and sync the file handle.
4. Rename the temp file over `{sessionId}.json`; same-directory rename is atomic on the target platforms used by this project.
5. Best-effort fsync the directory after rename on platforms where Node supports it.
6. If any step fails, remove only the temp file created for that write and leave the previous `.json` untouched.

Read behavior:

- Ignore `*.tmp` files during normal reads and save-list scans.
- If the main `.json` is malformed, return a corruption error and do not auto-delete it.
- Future recovery can optionally look for `{sessionId}.bak`, but backup rotation should be a separate step.

## Concurrent Access Protection

The Express server can receive overlapping requests for the same `sessionId`. S38.2 adds:

- In-process queue: `mutateSession(sessionId, mutator)` serializes read-modify-write work per session id.
- Optimistic revision: every persisted record has `revision`. A mutation reads revision `N` and writes only if the latest record is still `N`, then persists `N + 1`.
- Route ownership: `POST /api/game/turn`, `POST /api/exam/question`, and `POST /api/exam/submit` now use `mutateSession()` for state-changing work.

Out of scope for the JSON hardening pass:

- Cross-process locks for multiple Node servers sharing one directory.
- Network filesystem semantics.
- Multi-session transactions.

If multi-process local serving becomes necessary before a database migration, use lock files with stale-lock detection or move directly to SQLite instead of relying on advisory in-memory queues.

## Save List And Cleanup

The read-only save-list API is implemented:

```text
GET /api/game/saves
```

Response:

```json
{
  "saves": [
    {
      "sessionId": "uuid",
      "storageSchemaVersion": 1,
      "revision": 2,
      "createdAt": "2026-05-06T00:00:00.000Z",
      "updatedAt": "2026-05-06T00:01:00.000Z",
      "playerName": "未定",
      "role": "scholar",
      "roleLabel": "书生",
      "dynasty": "明",
      "year": 1644,
      "month": 1,
      "turnCount": 0,
      "examRank": null,
      "palaceRank": null,
      "officeTitle": null,
      "summary": "寒窗士子"
    }
  ],
  "skipped": []
}
```

Save-list rules:

- List only `.json` session records with safe UUID filenames.
- Sort by `updatedAt` descending, then `createdAt`, then filename for stable output.
- Use rebuilt envelope metadata; legacy raw saves may be listed through a safe in-memory metadata rebuild.
- Do not expose hidden relationship ids, provider configuration, raw prompts, local file paths, or full `worldState` in the list payload.
- Browser restore still supports direct restore of the last `localStorage` id, while the S38.3 save picker lets players choose from the redacted list on the start page or through the in-game save modal.

Cleanup policy:

- Keep manual/player saves indefinitely by default.
- Browser smoke and tests should continue deleting their known session ids explicitly.
- `cleanupSessionTempFiles()` can remove stale `*.tmp` files older than a configurable age.
- Never delete malformed `.json` files automatically; move them to a quarantine folder only after a future explicit admin command or script.

## Future Database Migration Path

Do not let route code depend on JSON-file details. The next migration slice should formalize the storage adapter boundary:

```javascript
{
  readSession(sessionId),
  writeSession(worldState, options),
  mutateSession(sessionId, mutator, options),
  listSessions(options),
  deleteSession(sessionId, options),
  cleanupSessionTempFiles(options)
}
```

Migration phases:

1. DONE: JSON envelope and atomic writes, with route-compatible `readSession()` and `writeSession()`.
2. DONE: Per-session mutation serialization plus revision tests.
3. DONE: Save-list API and explicit temp-file cleanup helper.
4. DONE: Browser save-list UI that consumes the redacted API while keeping last-session auto-restore compatibility.
5. NEXT: Storage adapter interface, with the current JSON implementation as the default adapter.
6. FUTURE: SQLite prototype for local development. Store one row per session with metadata columns, `revision`, timestamps, and a JSON `world_state` payload.
7. FUTURE: Export/import tooling from JSON envelope files to database rows and back.
8. FUTURE: Optional hosted database adapter only after local SQLite proves the contract.

The first database candidate should be SQLite because it improves atomicity and concurrent local access without requiring a service, credentials, or a hosted dependency. PostgreSQL or another hosted database can follow only when accounts, multiplayer, or remote save sync become product requirements.
