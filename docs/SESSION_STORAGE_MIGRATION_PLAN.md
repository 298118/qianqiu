# Session Storage Migration Plan

S38.2 is a planning step. It does not change runtime behavior yet. The current implementation remains the simple local JSON store in `src/storage/sessionStore.js`, but future storage work should follow this plan so JSON saves can evolve safely before any database move.

## Current Baseline

- Sessions are raw `worldState` JSON files at `data/sessions/{sessionId}.json`.
- `sessionId` is generated by `crypto.randomUUID()` and must match the UUID-like safe pattern in `sessionStore.js` before any path is built.
- `writeSession(worldState)` creates `data/sessions/` and writes formatted JSON directly with `fs.writeFile()`.
- `readSession(sessionId)` reads and parses the raw JSON file, returning `404` for missing safe ids and `400` for unsafe ids.
- Routes normalize older feature slices after reads with helpers such as `ensureRelationshipLedger()`, `ensureExamCalendarState()`, `ensureLongTermEventState()`, `ensureOfficialCareerState()`, and `ensureRoleWorldCouplingState()`.
- Browser restore currently stores only `qianqiu.sessionId` in `localStorage`; there is no server-side save list API yet.

Known gaps:

- No top-level session schema version or migration ledger.
- No atomic write protocol; a process crash during `writeFile()` can leave a partial JSON save.
- No in-process per-session write serialization or optimistic revision check.
- No save-list metadata, cleanup policy, backup/export shape, or database adapter boundary.

## Target Invariants

Storage changes must preserve these project invariants:

- `npm install && npm start` remains enough for local Mock-mode play.
- Real model providers remain optional and migrations never call providers.
- The server remains the authority for state boundaries, promotions, anti-cheat penalties, event queues, and persistence.
- The complete scholar path remains playable: scholar -> child exam -> provincial exam -> metropolitan exam -> palace exam -> official.
- `data/sessions/*.json`, `.env`, and provider keys stay out of Git.
- Legacy JSON saves should either migrate in memory and save back safely, or fail with an actionable error that does not corrupt the original file.

## Session Schema Versioning

Introduce a storage envelope before the first behavior-changing migration:

```json
{
  "storageSchemaVersion": 1,
  "sessionId": "uuid",
  "createdAt": "2026-05-06T00:00:00.000Z",
  "updatedAt": "2026-05-06T00:00:00.000Z",
  "revision": 1,
  "metadata": {
    "playerName": "未定",
    "role": "scholar",
    "roleLabel": "书生",
    "dynasty": "明",
    "year": 1644,
    "month": 1,
    "turnCount": 0,
    "examRank": null,
    "officeTitle": null,
    "summary": "寒窗士子"
  },
  "worldState": {}
}
```

Compatibility rules:

- Raw legacy files are treated as storage schema `0`, where the entire parsed object is the `worldState`.
- `worldState.sessionId` remains required and must match the file id. Envelope `sessionId` must match both.
- Feature-level `schemaVersion` fields such as `examCalendar.schemaVersion` and `officialCareer.schemaVersion` stay inside `worldState`; they are not replacements for the storage envelope version.
- Migrations must be pure functions: `migrateSessionRecord(record) -> { record, appliedMigrations }`.
- Migrations must clamp or normalize only through existing server-owned helpers where possible, then rebuild `metadata` from the resulting `worldState`.
- Unknown future `storageSchemaVersion` values should return a clear read error instead of silently downgrading.

Recommended migration order:

1. Add `sessionRecord` helpers that can wrap legacy raw saves without changing route payloads.
2. Add tests for raw legacy read, envelope read, mismatched ids, unsupported future version, and metadata rebuilding.
3. Switch `writeSession()` to write the envelope while keeping `readSession()` returning `worldState` for existing route compatibility.
4. Add explicit migration notes to `docs/SHARED_CONTEXT.md` and this plan each time the envelope version changes.

## Atomic Writes

JSON storage should move from direct overwrite to a temp-write-and-rename protocol:

1. Serialize the full session record before touching the existing file.
2. Write to a unique temp file in `data/sessions/`, for example `{sessionId}.{pid}.{timestamp}.tmp`.
3. Open the temp file, write UTF-8 JSON with a trailing newline, and `fsync` the file handle when available.
4. Rename the temp file over `{sessionId}.json`; same-directory rename is atomic on the target platforms used by this project.
5. Best-effort `fsync` the directory after rename on platforms where Node supports it.
6. If any step fails, remove only the temp file created for that write and leave the previous `.json` untouched.

Read behavior:

- Ignore `*.tmp` files during normal reads and save-list scans.
- If the main `.json` is malformed, return a corruption error and do not auto-delete it.
- Future recovery can optionally look for `{sessionId}.bak`, but backup rotation should be a separate step.

## Concurrent Access Protection

The current Express server can receive overlapping requests for the same `sessionId`. Future storage hardening should add two layers:

- In-process queue: serialize mutations per session id inside `sessionStore` or a small `sessionLocks` helper. `POST /api/game/turn`, `POST /api/exam/question`, and `POST /api/exam/submit` should not read-modify-write the same session concurrently.
- Optimistic revision: every persisted record has `revision`. A mutation reads revision `N` and writes only if the latest record is still `N`, then persists `N + 1`. If the revision changed, retry from the latest state or return `409` for non-idempotent calls.

Out of scope for the first JSON hardening pass:

- Cross-process locks for multiple Node servers sharing one directory.
- Network filesystem semantics.
- Multi-session transactions.

If multi-process local serving becomes necessary before a database migration, use lock files with stale-lock detection or move directly to SQLite instead of relying on advisory in-memory queues.

## Save List And Cleanup

Add a read-only save-list API after the envelope exists:

```text
GET /api/game/saves
```

Suggested response:

```json
{
  "saves": [
    {
      "sessionId": "uuid",
      "playerName": "未定",
      "role": "scholar",
      "roleLabel": "书生",
      "dynasty": "明",
      "year": 1644,
      "month": 1,
      "turnCount": 0,
      "examRank": null,
      "officeTitle": null,
      "updatedAt": "2026-05-06T00:00:00.000Z",
      "summary": "寒窗士子"
    }
  ],
  "skipped": []
}
```

Save-list rules:

- List only valid `.json` session records with safe UUID filenames.
- Sort by `updatedAt` descending, then `createdAt`, then filename for stable output.
- Use envelope `metadata`; legacy raw saves may be listed only after a safe in-memory metadata rebuild.
- Do not expose hidden relationship ids, provider configuration, raw prompts, or full `worldState` in the list payload.
- Browser restore should eventually prefer this list over a single `localStorage` id, while still supporting direct restore of the last id for compatibility.

Cleanup policy:

- Keep manual/player saves indefinitely by default.
- Browser smoke and tests should continue deleting their known session ids explicitly.
- Add an optional cleanup script before adding automatic server cleanup. The script can remove stale smoke/test saves older than a configurable age and stale `*.tmp` files.
- Never delete malformed `.json` files automatically; move them to a quarantine folder only after a future explicit admin command or script.

## Future Database Migration Path

Do not let route code depend on JSON-file details. Introduce a storage adapter boundary first:

```javascript
{
  readSession(sessionId),
  writeSession(worldState, options),
  mutateSession(sessionId, mutator, options),
  listSessions(options),
  deleteSession(sessionId, options)
}
```

Migration phases:

1. JSON envelope and atomic writes, with route-compatible `readSession()` and `writeSession()`.
2. Per-session mutation serialization plus revision tests.
3. Save-list API and optional cleanup script.
4. Storage adapter interface, with the current JSON implementation as the default adapter.
5. SQLite prototype for local development. Store one row per session with metadata columns, `revision`, timestamps, and a JSON `world_state` payload.
6. Export/import tooling from JSON envelope files to database rows and back.
7. Optional hosted database adapter only after local SQLite proves the contract.

The first database candidate should be SQLite because it improves atomicity and concurrent local access without requiring a service, credentials, or a hosted dependency. PostgreSQL or another hosted database can follow only when accounts, multiplayer, or remote save sync become product requirements.

## Verification Expectations

When S38.2 turns into implementation work, focused coverage should include:

- `node --test test/sessionStore.test.js`
- route tests for overlapping turn/exam mutations once `mutateSession()` exists
- legacy raw-save fixture migration tests
- malformed JSON and unsupported future-version read tests
- save-list sorting/redaction tests
- `npm test`
- `npm run smoke:browser` after any route/browser restore behavior changes

Documentation-only changes to this plan can be verified with `git diff --check` and UTF-8 readability checks.
